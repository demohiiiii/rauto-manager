## Brand & Style

Rauto Manager is a self-hosted **control plane for network-automation agents** — the kind of tool an SRE keeps open on a second monitor all day. Its design brief is the opposite of consumer playfulness: every surface should feel as if it were lifted straight out of a developer terminal and pressed flat into a modern web dashboard. The product's visual DNA is therefore a deliberate fusion of **shadcn/Radix minimalism** with **command-line nostalgia**.

The personality lands somewhere between *clinical instrument panel* and *hacker-room console*. Light mode is quiet, monochrome and clerical — it gets out of the way so operators can read dense status grids, IP tables, and JSON payloads. Dark mode flips the emotional temperature: a near-black deep-navy canvas, neon mint-cyan primary accents, faint CRT scanlines, and low-opacity glows around active nodes and edges. Both modes share the same structural skeleton and component language; only the "lighting" changes.

Two signature details carry the brand across every screen:

1. **JetBrains Mono as the global UI typeface.** Not just for code — for *everything*. Nav labels, page titles, stat values, button copy, input placeholders. This is the strongest single visual decision in the system and it is deliberate. It immediately signals "engineer tool" before a single pixel of chrome has rendered.
2. **The `>_` terminal prompt glyph.** It prefixes dialog titles, section headers, and task rows (e.g. `>_ Register rauto Agent`, `>_ [tasks.dispatchType.exec] show version`). It functions as the product's de-facto logotype-in-miniature and ties the web UI back to the CLI heritage of the underlying `rauto` engine.

The overall emotional target is **quiet confidence under load** — a UI that stays composed when an agent goes offline at 3 a.m. and whose visual noise floor is low enough that a red badge or a pulsing dot is instantly legible against it.

## Colors

The color system is built around a **near-monochrome spine** with strictly semantic accents. There is no decorative color — every chromatic pixel communicates state.

**Light mode (default, shown in product screenshots)** is a warm-neutral greyscale:

- **Canvas.** An almost-white `#fafafa` page background with pure white `#ffffff` cards floating on top. The one-tone difference is the only thing separating "page" from "surface," which keeps the UI feeling printed rather than layered.
- **Primary.** A near-black `#18181b` used sparingly — for active navigation pills, primary buttons, avatars, the product-mark tile, and the `ring` focus color. When it appears, it *reads* as ink, not as color. Primary buttons and active nav items get a subtle shadow tinted with the primary (`rgba(24, 24, 27, 0.24)`) so they feel gently stamped into the surface.
- **Chrome greys.** A tight ladder — `#f4f4f5` (muted), `#eeeeef` (accent/hover), `#e8e8ea` (secondary), `#dedee0` (border/input). Hover states almost always move one step up this ladder rather than introducing new hues.
- **Semantic set.** Success green (`#22c55e` with `#dcfce7` fill), destructive red (`#ef4444` with `#fee2e2` fill), warning amber (`#f59e0b` with `#fef3c7` fill), info blue (`#3b82f6` with `#dbeafe` fill). Semantic colors appear almost exclusively as *low-saturation tinted fills with strong-contrast foregrounds* inside badges, status pills, and notification rails — never as large solid blocks.

**Dark mode** is where the brand personality is dialed up. The canvas drops to a deep midnight `#04060d` — it is explicitly cooler and more saturated than black, closer to a server-rack LED glow than to pure darkness. Surfaces layer upward in cool navy steps (`#0b111e` → `#141c29` → `#1a2336`). The primary color switches from ink to **neon mint-cyan `#00ffe0`**, which is reserved for primary buttons, focus rings, selected flow-graph handles, and active scan-like indicators. Selected nodes and focused elements gain a layered cyan **"halo" shadow** — `0 0 0 1px rgba(34, 211, 238, 0.58)` on top of `0 0 0 5px rgba(34, 211, 238, 0.16)` — that reads as physical light bleeding through glass, not as a stroke.

A **five-way accent swap** (neutral, blue, green, purple, orange, rose) is exposed as a user preference. The chosen accent replaces the primary and focus-ring colors only; greys, semantics, and structural tokens do not move. This lets teams personalise the console without breaking recognisability.

Text contrast is enforced conservatively: body copy sits at `#09090b` on light surfaces and `#f8fafc` on dark; secondary copy uses `#71717a` / `#94a3b8`. There are no low-contrast greys for aesthetic effect — this is a tool whose readability is life-support.

## Typography

Typography is the single most identifying design move. **JetBrains Mono** is the sole UI typeface across the entire application, with system-monospace fallbacks. Every weight of hierarchy is expressed through size and weight, never through typeface change.

- **Display / Headline.** Used once per page, at the top. Heavy (600–700), tight tracking, slight negative letter-spacing at display sizes to counteract the looseness that monospace type has at large scale. Page titles like `Dashboard`, `Agent Management`, `Task Orchestration` sit at the display-md tier (30px / 700).
- **Title tier.** Card titles, dialog titles, section headers. 16–20px / 600. Dialog titles are frequently prefixed with `>_` followed by a space — the glyph is part of the type block, not an icon, so it inherits weight and tracking from the title.
- **Body & label.** Default reading size is 14px (`body-md`) with a comfortable 20px line-height. Form labels use 14px / 500. Small metadata (timestamps, captions, "n total", "created at") drop to 12–13px at 400–500 weight, in the muted-foreground grey.
- **Code & terminal output.** The same typeface, but inside dedicated terminal blocks (dark surface `#09090b`, light text `#e2e2e2`) at 12px / 400. Because the UI and the terminal share a typeface, copy-pasted CLI output inside a modal visually blends into the surrounding chrome — a deliberate continuity, not an accident.

The monospace grid creates a strong vertical rhythm: numerals align perfectly across stat cards, IP addresses and hostnames wrap predictably, and long command strings break at natural column boundaries. This is why the design tolerates — even requires — generous side padding inside cards.

## Layout & Spacing

The layout is a classic **two-pane admin console**: a persistent left sidebar, a sticky top header, and a scrollable main canvas.

- **8-point grid.** All dimensions derive from a 4/8/12/16/24/32/40/64-px ladder. Card internal padding is `24px` on all sides, the grid gutter between cards is `16px`, and the spacing between major page sections is `24px` (`space-y-6`).
- **Sidebar.** `256px` wide expanded, `76px` collapsed. The width transition is animated over 300ms with a standard ease, which is slow enough to feel intentional but fast enough not to interrupt workflow. Collapsed mode falls back to icon-only nav with tooltips on the right side.
- **Header.** `64px` tall, sticky, translucent (`rgba(255, 255, 255, 0.95)` with `backdrop-filter: blur(12px)`). Contains a left-biased search bar that occupies up to a 576px max-width, and a right-biased cluster of theme toggle, notification bell with count badge, and the "System Normal" pulse-dot pill.
- **Page canvas.** `24px` outer margin. Dashboards use a 4-up stat-card row (`grid-cols-4` on `lg`, `grid-cols-2` on `md`, stacked on mobile), followed by asymmetric 2-column blocks (usually 2/3 + 1/3) for activity feeds and quick-action panels.
- **Dialogs.** `max-width: 512px` centered, `24px` padding, with internal `16px` vertical rhythm between form groups. They overlay the page on an `rgba(0, 0, 0, 0.8)` scrim — darker than typical shadcn defaults, which reinforces the "focus mode" feel when a task is being composed.
- **White space strategy.** Density is moderate-high by design. Operators read these screens for long periods, so the spacing errs slightly tighter than a marketing dashboard would — but cards always breathe at `24px` internally so nothing crowds the border.

## Elevation & Depth

Depth is expressed through **border + shadow pairs**, not through layered blurs or transparency gradients. The elevation ladder has five steps:

- **e0 — Page canvas.** Flat, no shadow. The near-white background is the datum.
- **e1 — Inputs, outline buttons.** A 1px outline in `border` grey plus a hair-thin shadow (`0 1px 2px rgba(15, 23, 42, 0.04)`). Reads as "inset into the surface."
- **e2 — Standard cards, primary buttons.** A soft double-layer shadow in the shadcn tradition, giving cards just enough lift to separate them from the canvas without implying they are modal.
- **e3 — Popovers, dropdowns, tooltips.** A noticeably wider and softer shadow so the floating layer reads as temporary.
- **e4 — Select menus and panel overlays.** Same family as e3 but with more spread for taller menus.
- **e5 — Dialogs.** A deep diffuse shadow (`0 25px 50px -12px rgba(15, 23, 42, 0.35)`) plus the dark scrim, reinforcing that the dialog is a modal context switch rather than a popover.

Two motion-linked elevation tokens add life:

- **`hover-lift`** — cards lift by `translateY(-4px)` with a matching shadow bloom. Used on clickable stat cards, agent tiles, and history rows.
- **`neon-glow-dark`** — in dark mode, selected nodes/edges gain the layered cyan halo described in Colors. This is the *only* place in the system where shadow is used expressively rather than functionally.

In dark mode the shadow tokens do most of their work via glow rather than shade: outlines grow a soft cyan bloom, and inner shadows drop to near-zero because the dark canvas already provides the separation that shadow normally provides on light backgrounds.

## Shapes

The shape language is **restrained, slightly softer than default shadcn**, and consistent across scales.

- **Base radius** is `6–8px` (`rounded-md` / `rounded-lg`). Inputs, buttons, select triggers, badges, menu items, and tooltip bubbles all use this tier. It's enough curvature to feel modern and tactile but not so much that it reads as "playful."
- **Cards and dialogs** step up to `12–14px` (`rounded-xl`). This is the hero radius — every major content container uses it, including stat cards, dashboard panels, and modal bodies.
- **Full-round** is reserved for status dots, avatar circles, the 1.5px "active" indicator pill next to the selected sidebar item, and animated pulse elements.
- **Square corners** (`rounded-none`) appear only inside terminal output blocks, where monospace grid alignment reads better without rounding at the character edges.
- **Icons.** Lucide at 2px stroke with rounded caps and joins, sized primarily at 16px (inline with body), 20px (nav / stat-card icons), and 24px (empty-state illustrations). No filled icons — the line language is uniform.

The shape system deliberately avoids the extremes: nothing is sharp and brutalist, and nothing is pill-soft or glassy. The corner radii feel like a technical instrument with its edges just barely chamfered.

## Motion

Motion is **purposeful and understated in light mode, expressive in dark mode**. The system ships a broad palette but applies it with restraint.

- **Entrance.** Pages fade-in (500ms, decelerate) on load; lists stagger at 100ms intervals (`stagger-1` through `stagger-4`). Dialogs scale-in from 95% to 100% in 300ms with simultaneous fade. Sidebar nav items slide-in from the left on first render.
- **Feedback.** Hovering a clickable card triggers `hover-lift` (300ms ease). Buttons and interactive icons get `hover-scale` (1.05 in 200ms). Refresh icons and loaders use standard 1s linear spin.
- **Ambient.** The "System Normal" dot uses a 2s pulse loop. Skeleton placeholders run a 2s horizontal shimmer. Nothing in the main canvas floats or bobs — ambient motion is reserved for status affordances so the eye can find them.
- **Signature (dark mode only).** Two effects give dark mode its distinctive "server-rack" atmosphere:
  - **`glow-pulse`** — a 2s opacity oscillation between 1.0 and 0.5 on selected flow-graph handles and neon indicators, mimicking a blinking LED.
  - **`scan-line`** — an 8s linear translate from top to bottom of the viewport used sparingly on specific panels (terminal output, live-execution timelines) to suggest a CRT sweep. The opacity stays low (6% tint) so it reads as atmosphere, not as noise.
- **Easing.** The global default is the shadcn-standard `cubic-bezier(0.4, 0, 0.2, 1)` at 150ms. Entrances use a decelerate curve; exits use accelerate; continuous loops are linear or sinusoidal.

Transitions are **applied globally to `color`, `background-color`, `border-color`, `fill`, `stroke`** at 150ms, so theme and hover color changes always cross-fade rather than snap. This is the smallest detail but the one that most consistently makes the UI feel polished.

## Components

### App Shell & Navigation

The sidebar is the product's anchor. In expanded state it shows an icon-plus-label pair per route inside a `12px`-padded rounded rail. The **active item** is the single strongest visual event on the screen: the background fills to the near-black primary, text and icon invert to white, the icon scales up by 10%, and a small right-side dot indicator confirms selection. Inactive items are pure muted-grey text with no background; hover nudges them to the pale-grey secondary fill. At the top, a 36px rounded-lg primary-filled tile holds the product mark (a `Server` glyph) next to the "Rauto Manager" title and an operational subtitle.

The sidebar itself is translucent (`bg-card/60` + `backdrop-blur-sm`) in both themes, so content scrolling behind it shows through as a subtle wash — a small concession to modern polish against the otherwise flat sheet metal of the UI.

The header reinforces the "operations console" framing: a search bar with a contextual terminal-cursor icon, a notification bell with a red count badge, a light/dark toggle, and — on the far right — a green pulse-dot pill reading "System Normal" that turns amber or red as aggregate health degrades. The pill is the at-a-glance heartbeat of the system.

### Cards & Stat Tiles

All primary content lives inside `rounded-xl` white cards with a 1px outline-variant border and the e2 shadow. Stat tiles on the dashboard follow a strict template: a small-caps label in muted foreground top-left, a matching Lucide icon top-right, a display-md numeric value on the left side of the body, and a single-line caption beneath it in caption-sized muted text. Clickable stat tiles gain the `hover-lift` behaviour and a 2px focus ring on keyboard navigation.

Activity rows and notification items follow a companion pattern: a small rounded-md icon chip on the left, a title-plus-caption stack in the middle, and a status badge pinned to the right. Notification items add a 3px left border in their semantic color (info blue, warning amber, success green) which serves as the fastest possible skim-scan indicator.

### Buttons

Five variants cover every interaction:

- **Primary** — filled near-black, white text, e2 shadow. Used once per dialog (the affirmative action) and sparingly in page headers (e.g., "Create Task").
- **Outline** — white background, 1px border, e1 shadow. The workhorse for secondary actions like "Refresh," "View Details," "Cancel."
- **Secondary** — pale-grey fill, no border. Used for tertiary actions inside dense toolbars.
- **Ghost** — transparent, hover fills to accent grey. Used for icon-only affordances in headers and tables.
- **Destructive** — filled red `#ef4444`. Used only for irreversible actions like "Delete Agent."

Sizes step at `sm: 32px / default: 36px / lg: 40px` heights, all at `rounded-md`. Buttons containing icons reserve a 16px leading icon with an 8px gap.

### Inputs & Forms

Form fields are uniformly `36px` tall, `rounded-md`, with a transparent background and a 1px border in the input-border grey. Focus raises a 1px ring in the primary color (or the user's chosen accent) — explicitly *not* a glowing box-shadow, because the product avoids visual "sparkle" in ordinary interactions. Placeholders sit in muted-foreground grey and always describe the expected format (`e.g., core-sw-01`, `e.g., 192.168.1.1`). Field labels use the label-md tier in on-surface ink, with required-field asterisks rendered in the destructive red.

Dispatch-type tiles inside the "Create Task" dialog are a special compound input: a horizontal row of equal-width rounded-md tiles each showing an icon over a label (`Single Command`, `Template`, `Transaction Block`, `Workflow`, `Orchestration`). The selected tile adopts the primary fill — the same treatment as the active sidebar item — so the same "inverted ink" visual idiom communicates "this is what's currently selected" everywhere in the product.

### Badges & Status Pills

Badges are the system's communication workhorse. They use a flat `rounded-md` shape (never full-round pills, which would feel too marketing-y) at 12px label-sm type with 2–10px padding. Four variants — default (near-black), success (green-on-tint), destructive (red fill), outline (ghost) — cover task status, agent state, dispatch type, and device reachability. Status pills in the header add a pulsing 8px dot on the leading edge and a tinted fill in the matching semantic color at 10% opacity. Red, amber, and green are the only hues that appear outside the monochrome spine, which is why they reliably catch the eye.

### Dialogs

Dialogs are the primary "do work" surface. They overlay the page with a darker-than-default scrim (80% black) and center a 512px max-width panel with 24px padding. Titles always start with the `>_` glyph. Bodies use a 16px vertical rhythm between form groups. Footers follow the shadcn convention: secondary action (Cancel) on the left, primary action on the right. Destructive confirmations flip this — the destructive button is always on the right and paired with a mandatory descriptive warning above.

### Terminal & Code Blocks

The terminal block is a direct quotation from a CLI: near-black `#09090b` background, light grey `#e2e2e2` text, 12px monospace, 16px padding, `rounded-md`. When the task output contains ANSI-style semantic coloring, the block uses matching bright variants — `#4ade80` for success lines, `#22d3ee` for info, `#fbbf24` for warnings — over the dark surface. This block is visually distinctive enough that users recognise it at a glance as "this is raw device output, not UI."

Inline command pills (used in documentation and parameter descriptions) use a pale-grey filled `rounded-sm` chip with a 1px border — the miniature cousin of the full terminal block.

### Workflow / Orchestration Canvas

The flow designer uses ReactFlow tuned to the product palette. The canvas background is the card surface color overlaid with a dotted grid at 18% opacity slate. Node cards mirror the main card system: white, `rounded-lg`, e2 shadow. Selection is the expressive moment — in light mode, a selected node gains a soft blue halo (`rgba(59, 130, 246, 0.32)` 1px inset + 5px 10% bleed + 18px 14% drop) and lifts 1px; in dark mode, the halo switches to the neon cyan and widens substantially, producing the "plugged-in and hot" glow that the rest of the UI only hints at. Edges are 2.25px slate lines that fatten to 2.75px with a soft drop-glow when selected.

### Notifications & Toasts

Toasts (via Sonner) drop in from the top-right at 300ms with a slight scale. They use the same card geometry — `rounded-lg`, e3 shadow, 16px padding — and adopt the semantic color on their left edge as a 3px border, matching the notification-panel item treatment. This keeps the same "colored rail" mental model whether the notification is ambient in the panel or transient as a toast.

### Empty & Loading States

Loading states use skeleton placeholders at the exact dimensions of the final content, with a 2s horizontal shimmer. Empty states pair a 24–32px Lucide line icon in muted-foreground with a single sentence of guidance in body-md and a primary button to resolve the empty state (e.g., "Register your first Agent").

## Design Intent Summary

Rauto Manager's design is asking you to believe two things simultaneously: that network automation is **serious infrastructure work** that deserves a clinical, text-dense, high-signal interface; and that the people doing that work **enjoy the aesthetic of the terminal** and want their management console to wink at that heritage rather than hide it. Every decision — the global monospace, the `>_` prompt on dialog titles, the near-black primary in light mode and neon cyan in dark, the scanline motion reserved for live-execution views, the flat semantic badges, the tight 8-point grid — compounds into a single impression: **a control plane built by operators, for operators, that happens to render in a browser.**
